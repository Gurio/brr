## Detailed Findings

### 1. Architecture Coherence ✓

**The dominion model is solid.** Three layers, not conflicting:

- **Dominion** (owned, durable, git-backed) — resident's workshop
- **KB** (shared, curated, reviewed) — project's blueprint catalog
- **Runtime** (ephemeral, per-thought) — interface to the process

The promotion bridge (dominion → kb) is clean: the resident promotes on its own initiative; curators govern the kb.

**Account scope is right.** Unifies:
- Resident memory (old `brr-home` role)
- Account registry (forge identity, repo list, default repo)
- Cross-repo plans and decision ledger
- Dispatch inbox for message-event routing

All without polluting the project repo or creating forge objects by default.

### 2. CS5-CS7 Implementation ✓

#### CS5 — Inter-run plan

**What it does:** The resident writes `plans/<repo-slug>/active.md` and/or `plans/_cross-repo/active.md` in the account dominion. On the next wake, `_build_inter_run_plan_block()` reads both, injects as "Active inter-run plan", and remains silent when no file exists.

**Why it matters:** Plans survive between wakes. A resident-left plan is the *only* perception the next wake has of what was intended.

**Implementation quality:** Path helpers in `account.py` are clear. Block builder is defensive (catches exceptions, returns empty string). Silent-when-absent means no constant tax.

**Tests:** `TestInterRunPlanInjection` has 6 tests — absent-when-no-file, injects-repo-plan, injects-cross-repo-plan, includes-both, absent-when-empty, rides-in-daemon-prompt. All passing. ✓

#### CS6 — Runner policy

**What it does:** The resident writes `runner-policy/<repo-slug>/policy.md` and/or `runner-policy/_account/policy.md`. On wake, `_build_runner_policy_block()` injects them as "Stored runner policy".

**Why it matters:** Standing preferences for runner/core selection. The resident can express policy once and it shapes every task without re-stating.

**Implementation quality:** Follows CS5's pattern exactly. Repo-level policy rendered first, account-level second (precedence/clarity).

**Tests:** `TestRunnerPolicyInjection` has 6 tests — same pattern as CS5. All passing. ✓

**What's deferred (correctly):** CS6b — the daemon-owned confirmation step that applies proposed policy changes without the resident silently rewriting its own selection policy. This requires an explicit proposal+confirm loop in the daemon (operator reviews, approves, daemon applies). The plan page correctly marks this as CS6b follow-on. ✓

#### CS7 — Decision ledger

**What it does:** The resident creates `ledger/decisions.md` in the account dominion. On wake, `_build_decision_ledger_block()` injects it as "Decision ledger" — a user-facing through-line.

**Why it matters:** `kb/log.md` is technical (resident-facing, dated narrative). The decision ledger is strategic (user-facing, decision+rationale, cross-run continuity).

**Implementation quality:** Path helper + block builder follow the proven pattern. Web-visible when account dominion has a remote (local-first by default, opt-in projection).

**Tests:** `TestDecisionLedgerInjection` has 4 tests — absent-when-no-file, injects-when-present, absent-when-empty, rides-in-daemon-prompt. All passing. ✓

**What exists:** New models + schema for `ActivityRecord` (run/activity tracking), new endpoint `/accounts/activity`.

**What's missing:** No database migration (relying on SQLAlchemy auto-create), no daemon integration to actually *populate* the activity records.

**Impact:** The service changes are **additive** (no breaking changes), but they're the *shadow* of CS2/CS3 (per-run state docs, activity tracking) without the *threaded integration* (daemon calling the activity API to persist run records).

**Recommendation:** These changes can merge now (backward-compatible), but the full integration (daemon → brnrd activity API) is a **follow-on piece**. The user correctly asks: "do we need brnrd service changes? I think we do, but because it is currently an upsun-backed project — in case we need it deployed, we either have to merge it, or make a new active environment."

**Decision needed:** Does brnrd ship now or does this land when the daemon learns to write activity records? The code is correct either way, but the deployment pathway is not spelled out.

## How It Feels to Work Inside This Shape

The resident now has:

1. **Three new control surfaces** that survive between wakes:
   - A *plan* (what I'm doing now)
   - *Policy* (how I like to run)
   - *Ledger* (what I decided and why)

2. **Perception=injection model**: None of these surfaces are polled on every wake; they're injected directly. When they don't exist, there's no noise (silent when absent).

3. **Account scope**: A resident's memory is not scattered across repos or hidden inside a repo branch; it's a coherent account-scoped workspace that can coordinate across multiple repos.

4. **Ownership + durability**: The dominion is owned (the resident reshapes it freely), durable (git-backed, survives reinstall), and inspectable (plain files in git, no secrets).

The composition is **tight**: none of the three surfaces (plan, policy, ledger) conflict; they all slot into the same injection point and render consistently. The resident reads their own context before the shared project history. The injection infrastructure is symmetric (three new block builders, all following the same error-safe pattern).

**The coherence checkpoint holds.** The context you wake into is legible and held together by design, not accident.
